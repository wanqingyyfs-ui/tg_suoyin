## 2. 项目定位

**TG 索引**是一个面向公开 Telegram 资源的索引、搜索和导航项目，历史项目名为 `rectg`，所以数据库文件仍保留名称 `data/rectg.db`。

项目由以下部分组成：

- SQLite 主数据库；
- Telegram 公开页面爬虫；
- 自动清洗和 24 个一级大类分类；
- 命令行搜索工具；
- Astro 静态网站；
- Telegram Bot 搜索与消息锚点索引；
- 浏览器本地管理后台；
- Windows 桌面控制中心；
- 广告管理；
- GitHub Actions Windows EXE 和静态前端自动构建发布。

当前仓库地址：

```text
https://github.com/wanqingyyfs-ui/tg_suoyin
```

不要继续使用历史仓库：

```text
wanqingyyfs-ui/rectg
```

### 2.1 项目边界

项目当前的数据职责是：

```text
tg_shaixuan 负责前置筛选、审核和淘汰
        ↓
tg_suoyin 导入已接受链接
        ↓
抓取公开元数据、分类、搜索和展示
```

`tg_suoyin` 当前不根据成人、博彩、灰产、政治、宗教等行业词执行内容删除或隐藏。`scripts/filter_rules.py` 仅保留为旧脚本兼容文件，不是当前内容过滤入口。

## 3. 当前技术栈

### 3.1 Python

- Python 脚本负责数据库、爬取、分类、导出、Bot、后台和控制中心业务。
- 普通脚本依赖来自 `requirements.txt`：
  - `requests`
  - `beautifulsoup4`
  - `lxml`
  - `opencc-python-reimplemented`
  - `emoji`
  - `markdown`
- Windows 控制中心额外依赖来自 `requirements-control-center.txt`：
  - `PySide6`
  - `PyInstaller`

### 3.2 前端

- Astro 5；
- Node.js + npm；
- `pinyin-match` 用于中文和拼音搜索匹配；
- `@vercel/analytics`；
- `@vercel/speed-insights`。

### 3.3 数据库

- SQLite；
- 主数据库固定为 `data/rectg.db`；
- 数据库文件本身被 Git 跟踪；
- WAL、SHM、journal 和备份文件不提交。

### 3.4 Telegram Bot

当前 Bot 没有使用 `python-telegram-bot` 框架。项目通过 `scripts/bot_api_client.py` 直接调用 Telegram Bot HTTP API。

## 4. 重要结论和唯一入口

### 4.1 Bot 唯一公开入口

Bot 的唯一启动入口是项目根目录：

```text

```

```text
bot_core.py
```

Telegram API 客户端是：

```text
scripts/bot_api_client.py
```

已删除的旧文件：

```text
scripts/bot.py
```

不要重新创建或运行 `scripts/bot.py`。

启动 Bot：

```powershell
python bot.py poll
```

或者：

```powershell
.\run_bot.ps1
```

不要直接使用 `python bot_core.py` 作为日常启动方式。`bot_core.py` 是底层模块，根目录 `bot.py` 会导入它并替换搜索、按钮、分页和 update 处理逻辑。

```text
control_center_app.py
```

## 7. 环境变量

复制示例文件：

```powershell
Copy-Item .env.example .env
```

`.env` 被 `.gitignore` 忽略，不允许提交。

当前支持：

```dotenv
SITE_URL=https://tg-suoyin.vercel.app
TELEGRAM_BOT_TOKEN=your_bot_token_here
BOT_REQUEST_TIMEOUT=30
BOT_POLLING_TIMEOUT=25
BOT_WEBHOOK_HOST=127.0.0.1
BOT_WEBHOOK_PORT=8899
BOT_WEBHOOK_SECRET=change_this_secret
ADMIN_HOST=127.0.0.1
ADMIN_PORT=8787
ADMIN_TOKEN=
FRONTEND_HOST=127.0.0.1
FRONTEND_PORT=4321
```

### 7.1 各变量作用

| 变量 | 当前作用 |
|---|---|
| `SITE_URL` | 生成 sitemap、robots 和站点 URL；默认 `https://tg-suoyin.vercel.app` |
| `TELEGRAM_BOT_TOKEN` | Bot API Token |
| `BOT_REQUEST_TIMEOUT` | Telegram 普通 API 请求超时 |
| `BOT_POLLING_TIMEOUT` | `getUpdates` polling 超时 |
| `BOT_WEBHOOK_HOST` | 本地 webhook HTTP 服务监听地址 |
| `BOT_WEBHOOK_PORT` | webhook HTTP 服务端口，默认 8899 |
| `BOT_WEBHOOK_SECRET` | Telegram webhook secret token 校验 |
| `ADMIN_HOST` | 浏览器管理后台监听地址 |
| `ADMIN_PORT` | 管理后台端口，默认 8787 |
| `ADMIN_TOKEN` | 可选后台访问 Token，留空表示不启用 |
| `FRONTEND_HOST` | 控制中心内置静态前端服务地址 |
| `FRONTEND_PORT` | 控制中心内置前端端口，默认 4321 |

控制中心“设置”页面会原子写入 `.env`，原文件存在时保存 `.env.bak`。日志会遮蔽已知 Token 和 Secret。

## 8. 数据库是唯一长期数据源

主数据源：

```text
data/rectg.db
```

README 不再作为资源数据源，也不要恢复旧的 `web/build-data.js` 数据链路。

### 8.1 `links` 表

作用：保存待抓取的公开 Telegram 链接。

主要字段：

- `id`
- `url`
- `username`
- `name`
- `type_hint`
- `created_at`
- `updated_at`

链接导入、手动添加先进入 `links`，爬虫再将公开元数据写入 `entries`。

### 8.2 `entries` 表

作用：保存频道、群组和机器人的公开元数据。

主要字段：

- `id`
- `telegram_id`
- `username`
- `url`
- `type`：`channel` / `group` / `bot`
- `title`
- `description`
- `clean_title`
- `clean_desc`
- `category`
- `avatar`
- `count`
- `last_active`
- `valid`
- `private`
- `keep`
- `filter_reason`
- `created_at`
- `updated_at`

消息监听初始化后还会补充：

- `listen_enabled`
- `listen_status`：`off` / `active` / `error` / `paused`
- `listen_error`
- `listen_checked_at`
- `last_indexed_message_id`

### 8.3 当前可见条件必须按实际代码理解

当前 `scripts/export_frontend_data.py` 和 `scripts/search_entries.py` 使用：

```sql
valid = 1 AND private = 0
```

它们当前没有把 `keep` 放进查询条件。

因此：

- `valid=0` 的资源不会进入前端导出和资源搜索；
- `private=1` 的资源不会进入前端导出和资源搜索；
- `keep` 字段仍在数据库、后台和部分统计中使用，但当前前端导出和 CLI/Bot 资源搜索不会仅因 `keep=0` 排除资源。

维护时不要继续沿用旧说明中“前端固定要求 `keep=1`”的结论。

### 8.4 `message_index` 表

作用：保存 Bot 收到并允许监听的 Telegram 消息锚点。

基础字段：

- `id`
- `entry_id`
- `chat_id`
- `message_id`
- `chat_username`
- `chat_title`
- `chat_type`
- `message_date`
- `link`
- `keywords`
- `created_at`
- `updated_at`

统一 Bot 运行时会补充：

- `text_preview`
- `media_type`
- `media_emoji`
- `media_meta`
- `index_source`

消息索引不会把整条原始消息作为独立全文库保存。`message_indexer.py` 会生成用于搜索的紧凑关键词袋；统一 Bot 额外保存最多 1000 字符的展示预览。

### 8.5 `listener_settings` 表

当前键：

```text
terminal_summary_interval_seconds
```

默认 300 秒，最小 30 秒，最大 86400 秒。

### 8.6 `ads` 表

主要字段：

- `id`
- `position`
- `title`
- `description`
- `url`
- `image_url`
- `sort_order`
- `enabled`
- `created_at`
- `updated_at`

当前确认使用的广告位置包括：

- `bot_search_inline`
- `home_top`
- `home_below_hero`

Bot 搜索最多读取 2 条已启用的 `bot_search_inline` 广告。

### 8.7 `categories` 表

管理后台可同步该表，但当前分类标准真正来源于：

```text
scripts/categories.py
```

## 10. 完整数据链路

### 10.1 候选链接导入到网站

```text
tg_shaixuan 导出 JSONL / JSON / CSV
        ↓
scripts/import_collected_links.py
        ↓
links 表
        ↓
scripts/crawl.py
        ↓
entries 表
        ↓
scripts/categorize.py
        ↓
scripts/export_frontend_data.py
        ↓
web/public/data.json
web/public/sitemap.xml
web/public/robots.txt
        ↓
Astro build
        ↓
web/dist/
        ↓
Vercel 或本地静态服务
```

### 10.2 Bot 消息索引链路

```text
频道/群组资源在 entries 中开启监听
        ↓
Bot 必须加入对应频道/群组并具备权限
        ↓
bot.py poll 或 webhook 接收 Telegram update
        ↓
普通频道/群组消息不自动回复
        ↓
message_indexer.py 检查 listen_enabled=1 且 listen_status=active
        ↓
生成关键词袋和消息锚点
        ↓
写入 message_index
        ↓
Bot 的“全部”或“消息”搜索可返回消息锚点
```

### 10.3 数据和构建文件关系

```text
data/rectg.db
   ├─ web/public/data.json
   ├─ web/public/sitemap.xml
   ├─ web/public/robots.txt
   └─ web/dist/ 全部静态页面
```

当数据库条目减少后重新导出和构建，`web/dist/p/<id>/index.html` 会大量删除。这可能是正常构建结果，也可能是误用了错误数据库。提交前必须先确认数据库资源数量。

## 11. 资源导入和抓取

### 11.1 手动添加单个资源

```powershell
python scripts\add_entry.py https://t.me/example
python scripts\add_entry.py @example --name "示例频道" --type channel
python scripts\add_entry.py example --type group
```

添加后立即抓取：

```powershell
python scripts\add_entry.py example --type channel --crawl
```

`--crawl` 实际执行：

```powershell
python scripts\crawl.py --new --no-active
```

`--keep` 只会在对应 `entries` 已存在时设置 `keep=1`。

### 11.2 批量导入筛选器结果

支持：

- `.jsonl`
- `.json`
- `.csv`

执行：

```powershell
python scripts\import_collected_links.py --file ..\tg_shaixuan\exports\tg_suoyin_links.jsonl
```

npm 入口：

```powershell
npm run import-collected -- --file ..\tg_shaixuan\exports\tg_suoyin_links.jsonl
```

只检查不写入：

```powershell
python scripts\import_collected_links.py --file input.jsonl --dry-run
```

该脚本只接受公开 `t.me` / `telegram.me` username 链接，会跳过私密邀请、消息链接、分享链接、代理链接等不支持形式。

### 11.3 爬取新链接

```powershell
python scripts\crawl.py --new
```

跳过 `/s/` 活跃信息抓取：

```powershell
python scripts\crawl.py --new --no-active
```

测试少量数据：

```powershell
python scripts\crawl.py --limit 5 --no-active
```

刷新默认 30 天前的旧数据：


## 14. Telegram Bot 当前功能

### 14.1 文件分工

| 文件 | 当前职责 |
|---|---|
| `bot.py` | 搜索交互、相关度、总数、模式按钮、分页、callback query、统一 update 覆盖 |
| `bot_core.py` | polling、webhook、消息媒体处理、消息索引、广告、群内 mention、统计和 CLI 子命令 |
| `scripts/bot_api_client.py` | Telegram API 请求、发送/编辑消息、callback 应答、webhook 管理 |
| `scripts/message_indexer.py` | 监听字段、message_index 表、关键词袋、消息链接和数据库操作 |

### 14.2 私聊搜索

用户可以直接发送关键词：

```text
AI
```

也可以：

```text
/search AI
/s AI
```

`/start` 和 `/help` 会返回使用提示。

除其他 `/命令` 外，普通私聊文本会作为搜索关键词。

### 14.3 群内搜索

Bot 只响应以下形式，并引用触发消息：

```text
@机器人用户名 关键词
```

mention 必须位于消息开头。

普通群消息不会触发搜索回复，但如果该群组已开启监听，消息可以进入消息索引。

### 14.4 搜索范围

默认“全部”模式包含：

- 频道；
- 群组；
- 机器人；
- 已建立索引的消息锚点。

### 14.5 搜索结果格式

标题行：

```text
🔎 搜索：欧美，总计128 条相关结果
```

结果保持：

```text
序号. emoji[可选媒体信息] 限长锚文本
```

示例：

```text
1. 📢 频道简介锚文本
2. 👥 群组简介锚文本
3. 🎬[03:18] 视频消息锚文本
4. 📚[pdf 12.35MB] 文件消息锚文本
```

锚文本最多使用 20 个字符，超出后加省略号。

每页：

```text
14 条
```

第二页编号从 15 开始，而不是重新从 1 开始。

### 14.6 当前按钮只有六个

第一行：

```text
全部 | 群频 | 消息 | 最新
```

第二行：

```text
上一页 | 下一页
```

没有其他搜索按钮。

按钮含义：

| 按钮 | 当前行为 |
|---|---|
| `全部` | 频道、群组、机器人和消息锚点综合相关度排序，返回第一页 |
| `群频` | 只搜索频道和群组，不包含机器人和消息，返回第一页 |
| `消息` | 只搜索 `message_index`，返回第一页 |
| `最新` | 保持当前模式，按时间优先排序，返回第一页 |
| `上一页` | 当前模式和排序下回到上一页 |
| `下一页` | 当前模式和排序下进入下一页 |

第一页继续点击“上一页”会提示：

```text
已经是第一页。
```

最后一页继续点击“下一页”会提示：

```text
已经是最后一页。
```

### 14.7 Bot 排序

相关度模式综合：

- 搜索词与标题/username 的精确和包含关系；
- 简介和分类；
- 消息关键词袋；
- 词元覆盖率；
- `search_entries.py` 返回的资源分数；
- 时间和稳定 ID 作为后续排序。

“最新”模式优先：

- 资源 `updated_at` / `created_at`；
- 消息 `message_date` / `updated_at` / `created_at`；
- 相关度作为次级排序。

### 14.8 callback 状态

当前按钮状态通过内存缓存保存：

- 查询 token 使用关键词 SHA-256 截断值；
- 缓存有效期 24 小时；
- 最多保存 2048 项；
- Bot 进程重启后旧消息按钮状态可能失效；
- 失效时会提示重新发送关键词搜索。

### 14.9 广告

搜索结果存在时，Bot 会在普通结果前读取：

```text
ads.position = bot_search_inline
ads.enabled = 1
```

最多显示 2 条，按 `sort_order`、`id` 排序。广告不计入“总计 N 条相关结果”，也不占普通结果的全局编号。

### 14.10 消息媒体索引

当前识别：

- 文本 `📃`
- 文档 `📚`
- 音频/语音 `🎧`
- 视频/视频消息/动画 `🎬`
- 图片 `📸`

当前媒体信息：

- 文档：扩展名和文件大小；
- 音频、语音、视频、视频消息、动画：时长；
- 图片：无额外尺寸文本。

无自身文本或 caption 的媒体：

- 如果回复了一条有文本的消息，会用被回复文本建立关键词索引；
- 锚点仍指向当前媒体消息；
- 如果自身和回复都没有文本，不建立索引。

### 14.11 polling

启动：

```powershell
python bot.py poll
```

默认会删除 webhook，但不丢弃 pending updates。

保留现有 webhook：

```powershell
python bot.py poll --keep-webhook
```

指定终端汇总间隔：

```powershell
python bot.py poll --summary-interval 300
```

终端会汇总：

- update 数量；
- 消息收录数量；
- 私聊搜索；
- 群内搜索；
- 客户回复；
- 忽略；
- 错误。

### 14.12 webhook

启动本地 webhook HTTP 服务：

```powershell
python bot.py webhook-server
```

自定义：

```powershell
python bot.py webhook-server --host 127.0.0.1 --port 8899 --secret your_secret
```

路由：

```text
POST /tg-webhook
GET  /healthz
```

设置 Telegram webhook：

```powershell
python bot.py set-webhook https://example.com/tg-webhook --secret your_secret
```

查看：

```powershell
python bot.py webhook-info
```

删除并切回 polling：

```powershell
python bot.py delete-webhook
```

### 14.13 一个 Token 只能有一个 polling 进程

同一 `TELEGRAM_BOT_TOKEN` 不要同时启动多个 polling 进程。

冲突通常显示：

```text
Conflict: terminated by other getUpdates request
```

检查：

```powershell
Get-Process python -ErrorAction SilentlyContinue
```

确认旧进程无用后停止：

```powershell
Stop-Process -Name python
```

## 22. GitHub Actions 构建和自动发布

工作流：

```text
.github/workflows/build-control-center.yml
```

触发条件：

- 手动触发；
- 对 `main` 的 PR，且控制中心、脚本、Bot、依赖或前端路径发生变化；
- push 到 `main`，且上述路径发生变化。

构建环境：

- `windows-latest`；
- Python 3.12；
- Node.js 22。

工作流会：

1. 构建便携控制中心；
2. 生成 ZIP SHA256；
3. 上传 `TG-Suoyin-Control-Center-Windows` artifact，保留 30 天；
4. 当源码 push 到 `main` 时，将正式 EXE、`CONTROL_CENTER_BUILD.txt` 和 `web/dist` 自动提交回 `main`。

因此一次功能 PR 合并后通常会出现两个提交：

```text
功能/源码提交
chore: publish Windows control center [skip ci]
```

这是当前正常发布流程。

GitHub Actions Bot 自己提交的 `[skip ci]` 发布提交不会再次递归构建。

## 23. Git 跟踪规则和本地脏文件

当前明确被 Git 跟踪：

- `data/rectg.db`；
- `web/public/data.json`；
- `web/public/sitemap.xml`；
- `web/public/robots.txt`；
- `web/dist/`；
- 根目录正式 EXE；
- `CONTROL_CENTER_BUILD.txt`。

当前明确忽略：

- `.env`；
- `.venv/`；
- `node_modules/`；
- `logs/`；
- SQLite WAL/SHM/journal；
- `data/backups/`；
- `data/control_center.json`；
- `data/control_center.lock`；
- 普通 `.bak` / `.backup`。

由于数据库和构建产物被跟踪，运行控制中心、后台、Bot、导出或构建后，`git status` 出现修改是可能的。

不要在没有审查的情况下执行：

```powershell
git add -A
```

尤其要检查：

```powershell
git diff --stat
git status --short
```

SQLite 是二进制文件，普通 `git diff` 无法显示内部记录变化。

## 24. 安全同步远端 main

### 24.1 工作区干净

```powershell
cd "D:\编程\tg_suoyin"
git status
git switch main
git pull --ff-only origin main
```

### 24.2 工作区有本地修改

先确认这些修改是否需要保留。

临时保存全部修改，包括未跟踪文件：

```powershell
git stash push -u -m "同步前本地修改"
git switch main
git pull --ff-only origin main
git stash pop
```

数据库二进制文件在 `stash pop` 时可能冲突。重要数据库必须先额外复制到项目目录外：

```powershell
Copy-Item data\rectg.db "D:\rectg-backup-$(Get-Date -Format yyyyMMdd-HHmmss).db"
```

### 24.3 仅恢复误生成的前端文件

确认不需要本地生成结果后：

```powershell
git restore -- web/public web/dist
```

### 24.4 恢复被误改的数据库

只有已经在项目外备份当前数据库，并明确要放弃本地数据库变化时，才执行：

```powershell
git restore -- data/rectg.db
```

## 28. 常见问题

### 28.1 Bot 启动后没有按钮

确认启动的是：

```powershell
python bot.py poll
```

不是旧入口或 `bot_core.py`。

### 28.2 Bot 按钮点击无反应

确认：

- Bot 进程仍在运行；
- `scripts/bot_api_client.py` 的 `ALLOWED_UPDATES` 包含 `callback_query`；
- 旧消息按钮状态没有因 Bot 重启而失效；
- 没有多个 polling 进程抢 update。

### 28.3 Bot 搜索有资源但网页没有

重新导出和构建：

```powershell
python scripts\export_frontend_data.py
npm --prefix web run build
```

### 28.4 网页有资源但 Bot “消息”没有

“消息”只查询 `message_index`。需要：

1. 对应频道或群组在 `entries` 中开启监听；
2. Bot 已加入并具备接收消息权限；
3. 统一 Bot 正在运行；
4. 新消息具有可索引文本或回复文本。

### 28.5 `git status` 出现成百上千个详情页删除

通常表示当前数据库导出的资源集合比原 `web/dist` 少。先比较数据库和备份数据数量，不要直接 `git add -A`。

### 28.6 `database is locked`

- 停止重复后台、Bot 或控制中心实例；
- 控制中心已启用 busy timeout 和 WAL，但长期写事务仍会冲突；
- 检查是否有遗留 Python 进程；
- 不要删除正在使用的 `-wal` 或 `-shm` 文件。

### 28.7 控制中心 EXE 和源码看起来不一致

检查：

```powershell
Get-Content .\CONTROL_CENTER_BUILD.txt
```

根目录 EXE 是 Actions 在源码 push 到 `main` 后自动构建并提交的。等待或同步后续 `[skip ci]` 发布提交。

### 28.8 分类显示旧名称

优先运行：

```powershell
npm run rebuild
```

不要以 `edit_category.py` 内嵌旧列表作为当前标准。

## 29. 当前维护原则

1. 主数据库是 `data/rectg.db`，不是 README。
2. 不恢复历史 `web/build-data.js` 数据源。
3. 不混用旧仓库 `wanqingyyfs-ui/rectg`。
4. 站点 URL 使用 `SITE_URL`，不要写死旧域名。
5. 当前分类标准统一由 `scripts/categories.py` 管理。
6. `tg_suoyin` 负责抓取、分类、搜索和展示，不负责行业内容筛选。
7. Bot 唯一公开入口是根目录 `bot.py`。
8. `bot_core.py` 是底层模块，不是第二套独立 Bot。
9. 不恢复已删除的 `scripts/bot.py`。
10. 消息索引不导出到静态网站。
11. 修改数据库和生成文件前先确认 Git 状态。
12. 删除资源、清空索引和全量重爬前必须备份数据库。
13. 不提交 `.env`、Token、Secret、日志和数据库备份。
14. 不在未审查差异时使用 `git add -A`。
15. 修改代码后必须完成与改动相关的语法、构建或运行测试。
16. 文档只记录已经实现的行为；行为改变后应同步更新本文档。

```powershell
cd "D:\编程\tg_suoyin"

git remote -v
```

然后：

9. 合并后注意 Actions 可能追加正式 EXE 和 `web/dist` 发布提交；
10. 最后指导本地使用 `git pull --ff-only origin main` 同步。
